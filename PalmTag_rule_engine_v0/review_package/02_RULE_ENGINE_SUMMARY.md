# 规则引擎说明

## 整体流程

```text
33 字段输入
-> 补齐缺失字段默认值
-> 8 母型评分
-> 选择主母型/副母型
-> 只在主母型下匹配人格候选
-> 计算候选人格分数
-> 易混人格分流
-> 组装最终结果和展示文案
```

## 33 字段如何进入规则引擎

输入可以来自手动 JSON，也可以来自视觉模型三次识别投票后的 `final_features`。字段缺失时由 `field_schema_33.json` 中的 `default_value` 自动补齐。

## 8 母型如何评分

母型评分规则来自 `mother_scoring.json`。每条规则根据字段值、运算符、权重计算分数。母型分数封顶 100。主母型必须满足核心字段支撑约束，低置信字段不能单独决定主母型。

## 36 人格如何匹配

人格规则来自 `persona_rules.json`。规则引擎先确定主母型，再只在该母型下的人格候选中评分。每个人格由若干字段规则组成，支持 required 条件、普通加分、负权重和组合条件。

## 易混人格如何分流

易混分流规则来自 `adjacent_resolver.json`。只有当 Top1/Top2 人格分数接近时才触发，不会无条件覆盖结果。分流过程写入 `resolver_trace`。

## 主母型核心字段支撑约束

当前约束来自 `scoring_constraints.json`：

- 主母型至少 2 个核心字段命中。
- 低置信字段不能单独决定主母型。
- 母型分数最高 100。

## V0.1 / V0.2 随机 JSON 测试

V0.1 技术闭环通过，但随机分布发现 P25/P29 不命中、M8 偏高、M7 偏低、`HEART_LINE_DEPTH` 影响过高。

V0.2 修正后，1000 组随机输入全部跑通，P01-P36 均可命中，M1-M8 分布不再严重失衡。V0.2 随机分布中 M8 为 15.9%，M7 为 6.2%，无 display_content 缺失，无主母型核心支撑不足。

## V0.9 / V0.10 边界修正

V0.9 重点修正：

- HEAD_LIFE_GAP 视觉判断长期偏低。
- 强线条允许打到 3。
- P31 不能成为 M1 默认出口。
- P32 必须有强行动/强生命线支撑。
- M8 不应只靠复杂度或单字段误触发。

V0.10 只做极小边界修正：

- `dayi-left` 从 P25 修回 P31。
- `grand-right` 从 P31 修回 P20。
- 保护 `lan-right/zheng-right -> P29` 和 `zheng-left -> P32`。
