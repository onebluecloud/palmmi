# Palmmi Stage 3 Decisions

## 决策记录

1. V4.2 是 Stage 3 唯一规则源。
2. Palmmi 是正式产品名，PalmTag 仅作为历史旧名保留在原报告引用中。
3. Stage 3 从“设计阶段”改为“基于 V4.2 的工程翻译阶段”。
4. 不让 VLM 直接输出人格，只让 VLM 输出掌纹特征 JSON。
5. 不重新设计权重，必须实现 V4.2 的 8 个母型 score 函数和双层评分流程。
6. 不使用 `perceptual_hash`，只使用 `file_hash`。
7. 必须增加 Prompt 实测调优阶段 3E-b。
8. 必须建立 JSON Schema 校验、字段白名单、枚举校验、降级和重试机制。
9. 必须建立变更请求机制，有理由变更时走受控流程，不允许静默改规则。
10. Stage 3 不修改 Stage 2 UI。

## Stage 3B 决策

1. 用户原图不直接进入 VLM。
2. 识别图最长边控制在 1280px - 1600px。
3. 预览图和识别图分离。
4. EXIF 必须去除。
5. MVP 优先支持 `jpg` / `jpeg` / `png` / `webp`，HEIC 作为兼容风险项单独记录。
6. 原图默认不长期保存。
7. 3B 不判断是否是手掌，只做文件和输入标准化。
8. 3C 才判断非手掌和质量问题。

## Stage 3C 决策

1. 3C 是完整 VLM 前的质量门控层。
2. 3C 不输出 V4.2 33字段。
3. 3C 不判断人格和母型。
4. 明显非手掌图不得进入完整 VLM。
5. `LOW_QUALITY_PASS` 可以进入后续识别，但必须降低后续置信度。
6. `RETRY_REQUIRED` 和 `REJECTED` 不得进入完整 VLM。
7. 轻量视觉预检必须短 prompt、纯 JSON、低成本。
8. 用户失败提示必须面向普通用户，不使用工程术语。
9. AI 生成手/假手只做风险标记和明显拒绝，不在 3C 做复杂鉴伪系统。
10. 阈值可以后续调整，但必须走 `CHANGE_REQUESTS.md`。

## Stage 3D 决策

1. D-3D-01：V4.2 是 33 字段体系唯一来源。
2. D-3D-02：字段常量名必须完全沿用 V4.2 大写字段名，不允许重命名。
3. D-3D-03：`FINGERTIP_SHAPE` 是低置信字段，不进入 VLM Prompt，不作为核心母型判定。
4. D-3D-04：Palmmi 工程内部使用标准化 33 字段层，VLM 原始 JSON 需要在后续阶段映射和校验。
5. D-3D-05：3D 不实现代码，只冻结字段工程规范。

## Stage 3E 决策

1. D-3E-01：Palmmi 内部标准化字段层只能包含 V4.2 的 33 个大写字段。
2. D-3E-02：所有字段值必须是数字枚举或 `null`，不允许字符串枚举。
3. D-3E-03：字段校验失败不能直接进入规则引擎。
4. D-3E-04：字段降级必须记录，不能伪装成原始高置信识别结果。
5. D-3E-05：连续 5 个字段降级触发重拍。
6. D-3E-06：低置信字段不能单独决定主母型，也不能作为主母型核心字段支撑。
7. D-3E-07：3E 只冻结 Schema 与校验策略，不实现规则引擎。

## Stage 3E-b 决策

1. D-3E-B-01：V4.2 Prompt 是来源版，不直接视为生产版。
2. D-3E-B-02：生产版 Prompt 必须经过 10-20 张图实测后冻结。
3. D-3E-B-03：VLM 只做视觉字段提取，不输出人格、母型、命运、健康、婚姻、寿命判断。
4. D-3E-B-04：`FINGERTIP_SHAPE` 不进入 VLM Prompt。
5. D-3E-B-05：`prompt_version` 必须进入后续缓存 key。
6. D-3E-B-06：Prompt 冻结后，后续修改必须走 `CHANGE_REQUESTS.md`。
7. D-3E-B-07：如果没有测试图片和 API 配置，本轮不伪造实测结果，只创建测试方案和报告模板。

## Stage 3F 决策

1. D-3F-01：Palmmi Stage 3 只使用 `file_hash`，不使用 `perceptual_hash`。
2. D-3F-02：禁止视觉相似图片缓存。
3. D-3F-03：缓存 key 必须包含 `model_version`、`prompt_version`、`schema_version`、`degradation_policy_version`、`rule_version`。
4. D-3F-04：`prompt_version` 变化必须生成新缓存 key。
5. D-3F-05：`schema_version` 变化必须生成新缓存 key。
6. D-3F-06：`rule_version` 变化必须生成新缓存 key。
7. D-3F-07：不长期保存用户原图，压缩识别图如需缓存必须设置过期时间。
8. D-3F-08：缓存策略只保证同一文件结果稳定，不保证视觉相似图片复用。
9. D-3F-09：3F 只冻结缓存与版本策略，不实现缓存代码。

## Stage 3G 决策

1. D-3G-01：V4.2 §4、§5、§7 是规则引擎唯一来源。
2. D-3G-02：8 母型评分函数不得重新设计。
3. D-3G-03：评分字段、阈值、权重系数必须完全沿用 V4.2。
4. D-3G-04：主母型必须满足至少 2 个核心字段支撑。
5. D-3G-05：辅助字段和低置信字段不能单独决定主母型。
6. D-3G-06：规则引擎必须采用双层评分结构，不允许简化为 33字段直接到36人格。
7. D-3G-07：MOTHER_TO_PERSONA 映射必须严格沿用 V4.2 附录 B。
8. D-3G-08：相邻人格区分规则必须严格沿用 V4.2 §7。
9. D-3G-09：3G 只冻结规则引擎工程规范，不实现代码。
