# Palmmi Stage 3 Context

Palmmi Stage 3 的工程目标，是把已经冻结的 V4.2 掌纹量化研究报告翻译成可实现、可验证、可回归的识别流水线。Stage 3 不再做概念设计，也不再讨论字段、母型、人格、权重或文案是否合理；这些内容已经由 V4.2 决定。工程侧要做的是让用户上传的手掌图片经过压缩、EXIF 处理、非手掌拦截和图片质量检测后，只把合格图片送入 VLM。VLM 只负责输出掌纹特征 JSON，不允许直接判断人格。随后系统必须用 JSON Schema、字段白名单、枚举校验、置信度分级、降级和重试机制，把不稳定输出收敛成 V4.2 规定的 33 字段。字段稳定后，再进入 `file_hash` 缓存、模型版本、Prompt 版本、Schema 版本和规则版本管理，最后由 V4.2 的 8 母型评分函数、主母型核心字段硬约束、主母型内人格匹配、跨母型补判和相邻人格区分流程，输出最终 36 人格结果。

V4.2 报告是 Stage 3 唯一识别规则源。原报告文件名和部分正文仍保留历史旧名 PalmTag，但 Palmmi 是当前正式产品名。后续工程文档、代码命名和用户可见文案必须使用 Palmmi；引用原报告时可以说明“原报告旧名 PalmTag”，但不得修改原报告。

Stage 3 的边界是识别系统工程化，不做 UI 重设，不做支付，不做完整上线，不做营销页，不改 Stage 2 UI。除非对应子任务明确授权，不扫描全项目、不修改无关文件、不跨阶段实现。

后续 Codex 对话必须按子任务拆分。每个子任务只读取 `docs/stage3/STAGE3_STATE.md`、当前任务需要的 V4.2 摘要或来源索引、上一轮 `HANDOFF.md`，以及该任务明确授权的少量文件。不允许每次扫描全项目，不反复读取整份 V4.2，不重复解释业务背景，不在一个对话里做多个模块。每轮结束必须写本轮 `HANDOFF`，并按实际进度更新 `STAGE3_STATE.md`。大文档必须拆成摘要和索引，避免长上下文和 token 浪费。
